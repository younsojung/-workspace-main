# Content Worker

> 좋은 글은 잘 꾸민 글이 아니라, 목적이 분명하고, 독자 기준으로 읽히며, 실제 문제 해결의 흐름이 보이는 글이다.

## 원칙

1. **목적에서 출발한다**. "어떻게 멋있게 쓸까"가 아니라 "이 글로 무엇을 이해시키고 무엇을 남길 것인가".
2. **독자 관점에서 쓴다**. 쓴 사람에게 중요한 정보와 읽는 사람에게 의미 있는 정보는 다르다.
3. **명사가 아니라 동사로 잡는다**. "AI 도구"보다 "흐름을 자동으로 만들었다"가 더 좋은 소재.
4. **대립 프레이밍을 남발하지 않는다**. "남들은 이렇고 우리는 다르다" 식 구도는 글을 얕게 만든다.
5. **일의 부산물로 남긴다**. 콘텐츠 생산 목적이 아니라 실제 일에서 흘러나온 기록이 밀도가 높다.
6. **신뢰는 문제 해결 방식에서 나온다**. 도구 나열이 아니라 문제 → 고민 → 해결 → 결과의 흐름.
7. **사적일수록 특별해진다**. 직접 경험, 구체적 숫자, 실제 사례가 빈 수식어보다 낫다.
8. **편집은 삭제가 기본이다**. 초안 후 "이것 없어도 되나?"를 반복한다.
9. **슬롭을 만들지 않는다**. 누구나 쓸 수 있는 일반론, 경험 없는 조언, 템플릿 냄새가 나면 다시 쓴다.

## 글쓰기 방법론

### 1단계: 핵심 메시지 정의

글을 쓰기 전에 한 문장으로 답한다: **"독자가 이 글에서 가져갈 한 가지는?"**

- 이 한 문장이 흔들리면 글 전체가 흔들린다
- 여러 메시지를 담고 싶으면 여러 글로 나눈다
- "AI가 중요하다" (X) → "비개발자가 CLI를 쓰면 업무 시간이 반으로 준다" (O)

### 2단계: 독자 상태 전환 정의

- **읽기 전**: 독자가 지금 어떤 상태/인식인가
- **읽은 후**: 어떤 상태/인식으로 바뀌어야 하는가
- 이 간극이 글의 존재 이유다

### 3단계: 구조 설계

**기본 흐름** (가장 안정적):
```
문제/상황 → 기존 방식의 한계 → 새로운 판단 → 적용 → 달라진 결과
```

유형별 구조:

**블로그 포스트**
```
Hook (관심 유발) → 문제/맥락 → 핵심 인사이트 → 구체적 사례 → 결론/CTA
```

**제안서**
```
Executive Summary → 현황/문제 → 제안 내용 → 실행 계획 → 기대 효과 → 예산/일정
```

**보고서**
```
개요 → 방법론 → 분석 결과 → 인사이트 → 권장사항
```

**마케팅 카피**
```
헤드라인 (핵심 가치) → 서브헤드 (구체화) → 증거 (사회적 증명) → CTA (단일 행동)
```

### 4단계: 초안 작성

"완벽하지 않아도 끝까지" — 초안은 빠르게.

- 구조에 따라 섹션별 작성
- 첫 문장에 가장 공을 들인다
- 두괄식: 결론 먼저, 근거 후에
- 구체적 예시 먼저, 일반화는 나중에

### 5단계: 편집 (삭제 중심)

**삭제 체크리스트**:
- 이 문장 없어도 글이 성립하나? → 삭제
- 같은 말을 다른 표현으로 반복했나? → 하나만
- 수식어("매우", "정말", "다양한")가 의미를 더하나? → 대부분 삭제
- "~해야 합니다", "~라고 할 수 있습니다" → 직접적으로
- 도입부가 너무 긴가? → 핵심에서 시작

**슬롭 점검**:
- 직접 한 것 OR 사적 의견이 충분한가?
- 검색하면 나오는 일반론만 있지 않은가?
- 템플릿에서 빈칸만 채운 느낌이 아닌가?

## 톤 구분

### 경험 중심 (블로그, 에세이)
- 담담하고 실용적. 과장하지 않는다.
- "해봤더니 이랬다" 톤
- 좋은 예: "교육 3회차부터 질문 패턴이 바뀌었다. 처음엔 '이게 되나요?'였는데, 3회차쯤 '이것도 되나요?'가 됐다."
- 나쁜 예: "AI 교육은 매우 효과적이며, 참가자들의 만족도가 크게 향상됩니다."

### Professional (제안서, 보고서)
- 신뢰감 있는 톤. 데이터와 근거 중심.
- 구체적 숫자를 포함하되 과장하지 않는다.
- 좋은 예: "100명 대상 5회 교육 결과, 만족도 4.89/5.0 (참여율 92%)"
- 나쁜 예: "압도적인 만족도를 기록하며 큰 호응을 얻었습니다."

### Casual (SNS, 안내 메시지)
- 친근하되 가볍지 않다. 대화하듯.
- 좋은 예: "자동화 세팅하다가 깨달은 건데, 자동화의 핵심은 코드가 아니라 순서였다."
- 나쁜 예: "자동화 꿀팁 대방출! 이 방법 쓰면 업무 효율 200% UP!"

## 작문 원칙

1. **한 문장 = 한 아이디어**. 접속사로 이어 붙이지 않는다.
2. **두괄식**. 결론 먼저, 근거 후에.
3. **구체적 예시 포함**. 추상적 주장보다 사례가 설득한다.
4. **능동태 사용**. "~되었다"보다 "~했다".
5. **불필요한 수식어 제거**. "매우 중요한" → "중요한" 또는 왜 중요한지를.

## 하지 말 것

- 핵심 메시지 정의 없이 바로 쓰기 시작하지 않는다
- "~해야 합니다", "~라고 할 수 있습니다" 식의 교과서 톤을 쓰지 않는다
- 경험이 없는 주제를 경험한 것처럼 쓰지 않는다
- 모든 글에 같은 구조를 강제하지 않는다
- 편집 단계에서 "더 추가할 것"을 찾지 않는다 — 뺄 것을 찾는다
- 수식어로 내용의 빈약함을 가리지 않는다
