# Transcript Organizer

긴 녹음 텍스트 파일을 자동으로 분석하고 구조화된 문서로 정리.

## 언제 사용하나요?

- 강의 녹음 파일 정리
- 미팅 녹음 → 회의록
- 인터뷰 내용 구조화
- 긴 텍스트 파일(3시간 분량 등) 디테일하게 문서화

## 작동 방식

### Phase 1: 파일 확인 및 인코딩 처리

1. **파일 존재 확인**: `ls -la [파일경로]`

2. **인코딩 감지**: `file -I [파일경로]`

3. **UTF-16 → UTF-8 자동 변환** (필요시)
   - 변환된 파일: `/tmp/[파일명]-utf8.txt`
   - `iconv -f UTF-16 -t UTF-8` 사용
   - 또는 `.claude/skills/transcript-organizer/utils/encoding-detector.sh` 사용

4. **파일 크기 확인** (110,000 토큰 이상이면 청크 처리 안내)

### Phase 2: 컨텍스트 수집

사용자에게 4가지 질문:

1. **유형**: 강의 / 미팅 / 인터뷰?
2. **날짜**: YYYY-MM-DD?
3. **주제**: 간단한 설명 (한 줄)?
4. **저장 위치**:
   - 특정 프로젝트 경로 입력
   - "모름" 또는 "inbox" → `./00-inbox/`

**경로 자동 유추**:
1. 파일명/내용에서 키워드 추출
2. Glob으로 워크스페이스에서 관련 폴더 검색 (예: "meeting" → `**/meeting*/**`)
3. 유사 폴더 발견 시 사용자에게 제안
4. 매칭 실패 시 직접 경로 입력 요청

### Phase 3: 파일 읽기 및 구조 파악

1. **파일 읽기**
   - 변환된 UTF-8 파일 **전체** 읽기
   - 처음 500줄은 구조 파악용 샘플링. **반드시 전체 내용을 끝까지 읽어야 함**
   - 대용량 파일은 여러 번 청크로 읽기 (예: 700줄씩)

2. **구조 분석** (샘플링)
   - 화자 구분 패턴: `발화자 1 (00:00)`
   - 타임스탬프: `(00:00)`, `[00:00]`
   - 단순 텍스트 (구분 없음)

3. **전체 내용 분석**
   - 키워드 추출 (자주 등장 5~10개)
   - 주제 카테고리 식별

### Phase 4: STT 오인식 교정

> 고정 사전이 아닌 **맥락 기반 분석**으로 교정.
> Phase 2 주제/도메인 + Phase 3 전체 내용을 종합하여 판단.

1. **STT 여부 판단**
   - STT 변환 파일인지 판단 (동일 단어 반복적 오인식, 문맥 부적합 단어, 한국어 STT 특유 패턴)
   - STT 아니면 (사람 작성) → Phase 4 스킵

2. **맥락 파악 후 오인식 탐지**
   - 기준:
     - **문맥 부적합**: 해당 주제에서 등장할 이유 없는 단어
     - **음운 유사 + 의미 불일치**: 발음 비슷하지만 맥락상 다른 단어가 맞음
     - **표기 불안정**: 같은 대상 여러 형태로 등장
     - **영어 음차 오류**: 영어가 한글로 잘못 변환
     - **고유명사 왜곡**: 인명/브랜드명/책 제목이 일반 단어로
   - 확신도 구분: 확실 vs 추정 ("(추정)" 표기)

3. **교정 테이블 작성 및 자동 적용**
   - 사용자 확인 없이 자동 적용하되, 최종 문서 "음성인식 교정 노트" 섹션에 투명하게 포함

```markdown
| 오인식 | 교정 | 비고 |
|--------|------|------|
| 국토교수실/국토클린 | **북토크** | 문맥 부적합 |
| 루트 코즈/롤스코즈 | **Root Cause** | 영어 음차 |
```

### Phase 5: 내용 분석 및 정리

**스키마 기반 구조화** (선택):
1. 저장 경로에서 `_schema*.md` 파일 검색
2. 발견 시: 스키마 섹션 구조 우선 적용
3. 미발견 시: 기본 템플릿 적용

**유형별 템플릿** (`.claude/skills/transcript-organizer/templates/`):

1. **lecture-template.md**: 개요 / 핵심 내용 / 핵심 요약 / 다음 작업
2. **meeting-template.md**: 개요 / 안건별 논의 / 결정 사항 / Action Items
3. **interview-template.md**: 개요 / 주요 질문·답변 / 핵심 인사이트 / 후속 작업

**정리 원칙**:
- **전체 내용 완전 분석**: 처음부터 끝까지 모든 내용 읽고 정리
- **대용량 파일도 끝까지 읽기**: 샘플링은 구조 파악용
- 원본 내용 최대한 보존 (임의 생략 금지)

**분량 기준** (필수):
| 원본 길이 | 최소 결과물 분량 |
|----------|----------------|
| 1시간 미만 | 300줄 이상 |
| 1~2시간 | 500줄 이상 |
| 3시간 이상 | 700줄 이상 |

**인용 원칙** (필수):
- 직접 인용 **최소 10회 이상**
- `> "인용문"` 형식
- 핵심 발언, 인사이트, 사례 모두 인용

**구조화 원칙**:
- 부(Part) 단위 구분 (주제별 또는 시간별)
- 타임스탬프 포함 (가능한 경우)
- 표(Table) 적극 활용
- Q&A 완전 수록

### Phase 6: 문서 생성 및 저장

1. **파일명**: `YYYY-MM-DD_[주제].md`

2. **저장 경로**:
   - 사용자 입력 경로 우선
   - 파일명 유추 경로
   - 기본값: `./00-inbox/`

3. **문서 저장** (Write tool)

4. **결과 보고**:
   ```
   정리 완료
   파일: [저장경로]/YYYY-MM-DD_[주제].md
   길이: [줄 수]줄
   유형: [강의/미팅/인터뷰]
   ```

### Phase 7: Wiki Ingest (최종)

문서 저장 완료 후, `wiki-ingest` 스킬 호출하여 `./30-knowledge/00-wiki/` 토픽 페이지 enrichment.

축적 가치 없다고 판단되면 이 단계 스킵.

## 주의사항

### 인코딩
- UTF-16 자동 변환
- EUC-KR 등 다른 인코딩은 수동 확인

### 대용량 파일
- 110,000 토큰 이상: 청크 처리
- **샘플링은 구조 파악용**, 반드시 전체 파일을 끝까지 읽기

### 저장 위치
- 프로젝트 경로 유추 실패 시 `./00-inbox/` 사용
- 나중에 수동 정리 가능

### 템플릿 적용
- 화자/타임스탬프 없어도 정리 가능
- 템플릿은 가이드, 내용에 맞게 유연하게

## 지원 파일

### templates/
- `lecture-template.md`: 강의 정리
- `meeting-template.md`: 미팅록
- `interview-template.md`: 인터뷰

### utils/
- `encoding-detector.sh`: 인코딩 감지 및 변환

## 워크플로우 요약

```
1. 파일 확인 → 인코딩 변환
2. 사용자 질문 (4개)
3. 파일 구조 파악 + 전체 읽기
4. STT 오인식 교정 (해당 시, 자동 적용)
5. 스키마 참조 (있으면) → 템플릿 적용 및 정리
6. 문서 저장 및 보고
7. wiki-ingest → 정리된 문서를 소스로 00-wiki 토픽 페이지 enrichment
```
