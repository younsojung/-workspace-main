# Dashboard Design Principles

> Claude 내부 참조용. 사용자에게 직접 보여주지 않음.
> Phase 3 화면 설계 시 이 원칙을 적용하여 View/차트/KPI를 제안.

## 1. 대시보드 유형 분류

| 유형 | 목적 | 갱신 주기 | 대상 | KPI 수 | 특성 |
|------|------|-----------|------|--------|------|
| 운영 | 일상 모니터링 + 즉시 대응 | 실시간~일간 | 현장 관리자 | 5-10 | 이상 탐지, 알림 중심 |
| 전략 | 방향 설정 + 투자 결정 | 월간~분기 | 경영진 | 3-5 | 트렌드, 목표 대비 |
| 전술 | 단기 실행 조정 | 일간~주간 | 팀장/중간관리자 | 5-7 | 액션 연결, 드릴다운 |
| 분석 | 원인 탐구 + 가설 검증 | 필요 시 | 분석가 | 다수 | 필터/세그먼트 풍부 |

## 2. 결정 우선 프레임워크

**규칙**: 모든 차트는 "어떤 결정을 돕는가?"에 답할 수 있어야 한다.
- 답이 없는 차트 → 제거
- "보면 좋은" 차트 → 제거 (허영 지표)
- 하나의 차트가 여러 결정을 돕는다면 → 분리 고려

**적용 방법**:
1. Phase 1에서 수집한 의사결정 목록을 기준으로
2. 각 차트가 어떤 의사결정에 매핑되는지 명시
3. 매핑 안 되는 차트는 제안하지 않음

## 3. 5초 룰

- **메인 인사이트를 5초 내에 파악**할 수 있어야 함
- KPI 카드: 3-7개 제한 (운영 대시보드는 최대 10개)
- 차트: 화면당 최대 5개
- **좌상단 = 최중요** (시선 흐름: Z패턴 또는 F패턴)
- 스크롤 없이 핵심을 볼 수 있어야 함

## 4. 차트 선택 가이드

| 데이터 관계 | 추천 차트 | 비고 |
|-------------|-----------|------|
| 시간별 변화/추세 | 꺾은선 차트 (Line) | X축=시간, 시리즈 3개 이하 |
| 카테고리별 비교 | 수직 막대 차트 (Bar) | 카테고리 7개 이하 |
| 부분-전체 비율 | 누적 막대 차트 (Stacked Bar) | 파이는 5개 미만일 때만 |
| 순위/랭킹 | 수평 막대 차트 (Horizontal Bar) | 자연스러운 순위 표현 |
| 목표 vs 실적 | 프로그레스 바 / 불릿 차트 | 달성률 직관적 표현 |
| 단일 KPI | 카드 (현재값 + 비교 + 방향) | 숫자 + 화살표 + 맥락 |
| 분포 | 히스토그램 / 박스플롯 | 분석 대시보드에서 주로 사용 |
| 상관관계 | 산점도 (Scatter) | 분석 대시보드에서만 |
| 지리적 | 맵 (Choropleth) | 지역별 데이터가 있을 때만 |

**파이 차트 사용 조건**:
- 카테고리 5개 미만
- 합이 100%인 경우에만
- 그 외에는 수평 막대 차트가 항상 더 나음

## 5. 안티패턴

| 안티패턴 | 문제 | 대안 |
|----------|------|------|
| 허영 지표 (Vanity Metrics) | 의사결정에 연결 안 됨 | 결정 우선 프레임워크 적용 |
| 파이차트 남용 | 비교 어려움, 5개 이상 무의미 | 수평 막대 차트 |
| 이중 Y축 | 시각적 혼란, 오해 유발 | 차트 2개로 분리 |
| 스크롤 대시보드 | 한 눈에 파악 불가 | 탭/View 분리, 화면당 5차트 |
| 맥락 없는 숫자 | "1,234"만으로는 좋은/나쁜 판단 불가 | 비교(전기/목표) + 방향 화살표 |
| 3D 차트 | 왜곡, 읽기 어려움 | 2D만 사용 |
| 무지개 색상 | 의미 없는 색상 사용 | 색상은 의미 있을 때만 (빨강=위험 등) |
| 과도한 소수점 | 불필요한 정밀도 | 의사결정에 필요한 수준만 |

## 6. KPI 카드 설계

**필수 구성요소**:
```
+---------------------------+
| 매출                  ^5% |
| ₩12,450,000              |
| 전월 대비 +₩590,000      |
| 목표 달성률 89%           |
+---------------------------+
```

1. **라벨**: KPI 이름 (명확, 약어 금지)
2. **현재값**: 큰 글씨, 포맷팅 (천 단위 구분, 통화 기호)
3. **비교**: 전기 대비 절대값 또는 %
4. **방향**: 화살표 + 색상 (초록=좋음, 빨강=나쁨 — 맥락에 따라)
5. **맥락**: 목표 대비, 전년 동기 등

**주의**: 비용 KPI는 증가=빨강(나쁨). 방향 색상을 KPI 의미에 맞게 설정.

## 7. 데이터-잉크 비율 (Tufte 원칙)

- **장식 최소**: 3D 효과, 그림자, 불필요한 격자선 제거
- **데이터 최대**: 잉크의 대부분이 데이터를 표현해야 함
- **색상은 의미 있을 때만**: 강조, 카테고리 구분, 상태 표시
- **기본 색상**: 중립 (회색, 짙은 회색)
- **강조 색상**: 1-2개만 (브랜드 컬러 또는 파란계열)
- **상태 색상**: 빨강(위험/하락), 초록(양호/상승), 노랑(주의)

## 8. 한국 맥락 고려사항

- **화면당 차트 최대 5개**: 한국 기업 문화에서 "많을수록 좋다"는 유혹 있지만, 5개 넘으면 아무도 안 봄
- **누구나 읽을 수 있는 라벨**: 영어 약어 대신 한글 사용 (Revenue → 매출)
- **결과지표보다 관리지표**: 이미 지나간 숫자(매출)보다 바꿀 수 있는 숫자(전환율, 응대 시간)를 더 크게
- **비교는 전월/전년**: 한국 기업은 MoM, YoY 비교를 선호
- **엑셀 친숙도**: 대시보드가 엑셀보다 나은 이유를 보여줘야 채택됨
- **보고서 겸용**: 경영진은 대시보드 캡처를 보고서에 붙임 → 캡처해도 읽히는 레이아웃

## 9. 데이터 투명성 (Data Provenance)

**원칙**: 모든 KPI와 차트는 "이 숫자가 어디서 왔는가?"에 답할 수 있어야 한다.

### 필수 표시 4항목

| 항목 | 설명 | 예시 |
|------|------|------|
| 출처 (Source) | 이 숫자의 원천 데이터 | "ERP 수주", "네이버 정산 리포트" |
| 계산 근거 (Formula) | 어떻게 계산되었는가 | "SUM(sale_amount) WHERE month=현재" |
| 갱신 시점 (Freshness) | 언제 마지막으로 업데이트되었는가 | "데이터 기준: 2026-02" |
| 추정치 여부 (Estimation) | 확정값인가 추정값인가 | "온라인 원가는 소비자가 기준 추정" |

### UI 적용 규칙

- **KPI 카드 하단**: 출처 각주 한 줄 (예: "출처: ERP 수주 + 3사 정산")
- **계산식**: 마우스 툴팁으로 표시 (기본 화면은 깔끔하게 유지)
- **추정치**: 값 앞에 `~` 표시 (예: ~32.4%)
- **갱신 시점**: 대시보드 header에 "데이터 기준: YYYY-MM" 표시
- **5초 룰 유지**: 근거 정보는 각주/툴팁/footer에 배치. 메인 화면의 가독성을 해치지 않는다

### 안티패턴 (추가)

| 안티패턴 | 문제 | 대안 |
|----------|------|------|
| 출처 없는 숫자 | "이 매출이 세전? 세후? 수수료 빠진 거?" 질문 발생 | KPI 하단에 출처 한 줄 |
| 추정치를 확정값처럼 표시 | 잘못된 의사결정 유발 | ~ 표시 + 툴팁에 추정 근거 |

## 10. AI 인사이트 제안 패널

**원칙**: 대시보드는 숫자를 보여주고, AI는 "생각해볼 지점"을 제안한다. 결론이 아니라 질문을 던진다.

### 설계 규칙

- **위치**: 각 View 하단 또는 사이드 패널. 차트/KPI 영역을 침범하지 않는다
- **톤**: 단정이 아닌 제안 ("~일 수 있습니다", "확인해볼 필요가 있습니다")
- **근거 필수**: 모든 인사이트에 참조 데이터 명시 ("오늘의집 매출 +23%, 마진율 -2.1%p 기반")
- **개수 제한**: View당 최대 3개. 많으면 노이즈
- **유형**: 이상 탐지(급변), 추세 경고(3개월 연속), 기회 발견(비중 vs 성과 불균형)

### 안티패턴

| 안티패턴 | 문제 | 대안 |
|----------|------|------|
| AI가 결론을 내림 | 사용자가 검증 없이 수용 | "~일 수 있습니다" 제안형 톤 |
| 근거 없는 인사이트 | 신뢰 하락 | 참조 숫자 + 출처 명시 |
| 인사이트 과다 | 정보 과부하, 5초 룰 위반 | View당 3개 이하 |
